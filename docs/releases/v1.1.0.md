# Squarebox v1.1.0 migration guide

v1.1 changes the lifecycle model as well as adding tools. Test an upgrade with
`v1.1.0-rc5` before promoting the Release used for important work.

## Before upgrading

1. Commit or back up Workspace changes.
2. Record the current image tag and `docker volume inspect squarebox-home` output.
3. Keep the Managed home unless you intentionally want to reset authentication,
   histories, assistants, and mise toolchains.
4. If the installation uses custom paths or resource names, retain the old
   installer/uninstaller until the v1.1 Install identity has been written.

## What changes on first v1.1 rebuild

- The installer writes an Install identity at
  `<SQUAREBOX_DIR>/.squarebox/install-state`, containing effective paths,
  runtime, resource names, host UID/GID, source ref, and image identity. POSIX
  adapters use mode 0600; native Windows inherits the current user's install
  directory ACL. Release pulls record an immutable digest, while source/edge
  builds record a local image ID/reference. State is parsed as data and never
  sourced as shell code.
- New resources receive ownership labels. Legacy resources are adopted only
  through the installer's explicit compatibility path.
- Git name/email moves to installation-private configuration. Host credential
  helpers, includes, signing configuration, and tokens are no longer mounted.
- Image-managed dotfiles refresh safely into the existing Managed home. A
  symlink destination is never followed and makes startup fail visibly until
  the unsafe path is resolved.
- Saved Box-tier Selections such as tmux, Zsh, and Fish are reconciled with the
  new Box filesystem instead of trusting the old setup-complete marker.
- Dev Containers use `/workspace`; their Selection state now lives with the
  mounted repository instead of an unrelated image directory.

Existing v1.0 installations have no Install identity. Their first v1.1 run
must use `--adopt` (PowerShell: `-Adopt`) after reviewing the detected checkout,
Box, and Managed-home volume. Adopted legacy volumes still require explicit
`--force`/`-Force` before purge because they lack original ownership labels.

On Windows, choose the lifecycle adapter during that adoption and keep using it
for rebuild and uninstall. Native PowerShell and Git Bash use the same
`FORMAT=1` field names, but native `C:\...` paths/PowerShell profiles and MSYS
paths/Bash profiles are deliberately adapter-specific and are not cross-consumed.
Native PowerShell exposes `%USERPROFILE%\.ssh` read-only; it does not forward
`SSH_AUTH_SOCK`. Git Bash provides the separate agent-forwarding path.

## User-visible changes

- Zoxide no longer replaces `cd`; use `z` and `zi`.
- GitHub Copilot launches as `copilot`; the deprecated
  `github-copilot-cli` package/command is not retained.
- Learn mode is not shipped in the default image. Existing learn progress and
  hook files are inert and may be removed after backup.
- `sqrbx-rebuild` normally pulls the current published Release and recreates
  the Box. Use the explicit build option only when you intend a local image build.
- `sqrbx-update --apply` updates only installed registered tools. Naming an
  absent tool explicitly requests installation.

## What survives Box replacement

| State | Owner | Result |
| --- | --- | --- |
| Workspace files | Host Workspace | Preserved |
| Shell history, auth, assistant data, mise toolchains | Managed home | Preserved |
| Optional Managed-home tools | Managed home | Preserved and explicitly updateable |
| tmux/Zsh/Fish APT packages | Box tier | Reconciled from Selection |
| Image-tier binaries | Candidate image | Replaced by the published digest |
| Manually installed, unselected APT packages | Disposable Box filesystem | Lost |

## Release-candidate installation

Use the Candidate's published tag, which resolves matching source and image
identity from its Release assets:

```bash
(
  set -euo pipefail
  tag=v1.1.0-rc5
  tmp=$(mktemp -d)
  trap 'rm -rf "$tmp"' EXIT
  curl -fsSLo "$tmp/install.sh" \
    "https://github.com/SquareWaveSystems/squarebox/releases/download/$tag/install.sh"
  curl -fsSLo "$tmp/SHA256SUMS" \
    "https://github.com/SquareWaveSystems/squarebox/releases/download/$tag/SHA256SUMS"
  expected=$(awk '$2 == "install.sh" { print $1 }' "$tmp/SHA256SUMS")
  if command -v sha256sum >/dev/null; then
    actual=$(sha256sum "$tmp/install.sh" | awk '{ print $1 }')
  else
    actual=$(shasum -a 256 "$tmp/install.sh" | awk '{ print $1 }')
  fi
  test -n "$expected" && test "$actual" = "$expected"
  SQUAREBOX_TAG="$tag" bash "$tmp/install.sh" --adopt
)
```

This deliberately downloads the Candidate's installer instead of invoking the
old adapter already present in a v1.0 checkout. Do not combine an RC image with
a stable checkout. Verify `release.json`, the MOTD version, and
`/usr/local/lib/squarebox/VERSION` agree after installation.

## Rollback

1. Stop the v1.1 Box; do not purge the Workspace or Managed home.
2. Re-run the previous published installer with its explicit tag/image.
3. If the old release cannot consume the new Install identity, use its saved
   launch command and leave the v1.1 state file intact for a later retry.
4. Report the failed migration with both versions, source SHA, image digest,
   runtime, OS, UID/GID, and setup output.

Rollback cannot restore manually installed packages from a discarded Box
filesystem. It does preserve host Workspace files and the Managed home unless
the operator explicitly purges them.
